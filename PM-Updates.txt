CHANGE 6.5: INVENTORY PROFILES & RESET CAPABILITY
--------------------------------------------------------------------------------

GOAL: Allow users to manage multiple inventory profiles per set, each tracking
      independent collections (e.g., "Kayleigh", "Stream Opening", "Test Run")
      with optional reset capability.

IMPLEMENTATION OVERVIEW:

**New Data Structures:**

1. **InventoryProfile Interface** (types.ts)
   ```typescript
   interface InventoryProfile {
     id: string;              // Unique identifier
     name: string;            // Display name (e.g., "Kayleigh")
     collection: CollectionItem[];
     packsOpened: number;
   }
   ```

2. **Modified SavedSet Interface** (types.ts)
   ```typescript
   interface SavedSet {
     profiles: InventoryProfile[];      // Multiple profiles per set
     activeProfileId: string | null;    // Currently selected profile
     // Legacy fields for migration compatibility:
     collection?: CollectionItem[];
     packsOpened?: number;
   }
   ```

**Migration Strategy:**

- Old Format: `{collection: [...], packsOpened: 5}`
- New Format: `{profiles: [{id: 'default', name: 'Default', ...}], activeProfileId: 'default'}`
- Automatic Migration: Detects old format → creates "Default" profile with existing data
- Applied on every load via `migrateToProfiles()` function
- Legacy fields preserved for backward compatibility

**Backend Functions Added (useSetManager.ts):**

1. **createProfile(setId, profileName)**
   - Creates new profile with unique ID
   - Initializes empty collection and 0 packs opened
   - Automatically sets as active profile
   - Saves to storage immediately

2. **setActiveProfile(setId, profileId)**
   - Switches the active profile for a set
   - Used when selecting from dropdown
   - Saves to storage immediately

3. **updateProfileCollection(setId, profileId, pack)**
   - Replaces old updateSetCollection()
   - Updates specific profile's collection
   - Merges cards, increments counts
   - Increments profile's packsOpened counter
   - Saves to storage immediately

4. **resetProfile(setId, profileId)**
   - Clears all cards from profile collection
   - Resets packsOpened counter to 0
   - Preserves profile name and ID
   - Saves to storage immediately

5. **deleteProfile(setId, profileId)**
   - Removes profile from profiles array
   - If active profile deleted, sets activeProfileId to remaining profile or null
   - Cannot delete if only one profile exists (UI enforced)
   - Saves to storage immediately

6. **activeProfile (exported state)**
   - Computed value: finds active profile from activeSet
   - Returns null if no active profile or no active set
   - Used by UI to display current profile data

CHANGE 06: INVENTORY PERSISTENCE & COLLECTION TRACKING
--------------------------------------------------------------------------------

GOAL: Verify and strengthen collection system to reliably persist across sessions

CODE REVIEW FINDINGS:

**Issues Found & Fixed:**

1. **Type Safety Issue - Set<number> vs Set<string>**
   - Problem: `usedCardNumbers` declared as Set<number> but Card.number is string
   - Fix: Changed to `Set<string>()` for type correctness
   - Impact: Ensures proper duplicate prevention logic
   - Location: utils/packSimulator.ts line 61

2. **Storage Efficiency Issue - Unnecessary Data Persisted**
   - Problem: rarityPools saved to localStorage (wasteful, rebuilt on load anyway)
   - Fix: Strip rarityPools before saving using object destructuring
   - Impact: Reduces localStorage size, cleaner data persistence
   - Location: hooks/useSetManager.ts saveToStorage()

3. **Debugging Enhancement**
   - Added console logging to updateSetCollection
   - Shows collection size and packs opened count
   - Helps verify persistence working correctly

VERIFICATION OF EXISTING LOGIC:

**Collection Update Flow:**
1. Pack opened with 11 cards
2. Each card checked against collection (number + finish match)
3. If exists: increment count
4. If new: add with count = 1
5. Collection sorted by card number
6. SavedSets state updated immutably
7. saveToStorage() called immediately
8. localStorage updated synchronously

**Persistence Flow:**
1. On app load: loadFromStorage() runs during useState initialization
2. Parses JSON from localStorage
3. Rebuilds rarityPools for each set
4. Returns complete SavedSet array with collections intact

**Collection Item Structure:**
```typescript
{
  number: string,    // Card number (e.g., "25")
  name: string,      // Card name
  rarity: Rarity,    // Common, Uncommon, Rare, V, VMAX, Secret Rare
  finish: FinishType, // Standard, Reverse Holo, Holo, Ultra Rare, Secret Rare
  count: number      // How many owned
}
```

Each card number + finish combination tracked separately (correct for collecting)

TESTING CONFIRMATION:
• saveToInventory toggle enabled by default
• Collection updates when packs opened with toggle ON
• localStorage writes happen synchronously after each pack
• Browser close/reopen restores collection via lazy initialization
• No race conditions due to explicit save pattern
